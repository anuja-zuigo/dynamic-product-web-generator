# Project Rules

1. Always ask WHY before implementing any feature.
2. Keep the MVP simple and focused on the problem statement.
3. Backend first, frontend second.
4. Test every feature before moving to the next.
5. One responsibility per file and per function.
6. Use only free and production-capable technologies.
7. Never over-engineer to impress; build what solves the problem.
8. Every Git commit must represent a completed, testable feature.
9. Document major architectural decisions in DECISIONS.md.
10. If a simpler solution exists without reducing value, choose the simpler solution.

# Responsive Design Rules

The application must follow a Responsive-First design approach.

Every page must be fully functional on:

- Mobile Phones
- Tablets
- Laptops
- Desktop Computers
- Large Displays

Design Guidelines

- Mobile-first responsive layouts
- Flexible Grid system
- Fluid containers
- Responsive typography
- Responsive spacing
- Responsive navigation
- No horizontal scrolling
- Images scale automatically
- Tables become responsive on smaller devices
- Forms remain usable on touch devices

Implementation Guidelines

- Use CSS Flexbox and CSS Grid.
- Use responsive breakpoints.
- Avoid fixed-width containers.
- Use relative units where appropriate.
- Maintain accessibility and readability on all screen sizes.

Every new screen must be tested on multiple viewport sizes before it is considered complete.
